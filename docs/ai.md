# デジタルメータ監視システム PoC 開発ルール（重要）

本プロジェクトは現在、PoC（デモ可能な最小システム）構築フェーズである。
Codexを含むAIは、以下の優先順位とルールに従って実装を行うこと。

---

## 開発の最優先目標

最優先目標：

**エッジ端末 → 推論 → DB保存 → Reactダッシュボード表示 → 閾値通知**

までの一連の流れを、ローカル環境で動作する状態にすること。

過剰な抽象化や将来最適化よりも、**まず確実に動くデモ** を優先する。

---

## システム構成（現フェーズ）

構成：
Edge（Webカメラ / 将来Raspberry Pi）
↓
Python Backend（推論 / API / DB保存）
↓
SQLite Database
↓
React Frontend（ダッシュボード）
↓
Slack通知（閾値超過）

---

## 推論インターフェース統一ルール

YOLO版・Qwen版など複数モデルを使用するため、推論呼び出しは以下の形式に統一すること：

```python
def infer(image_path: str) -> dict:
    return {
        "meter_id": str,
        "value": float,
        "confidence": float,
        "model": str
    }
````

この形式を変更してはいけない。

---

## DB保存ルール

推論結果は必ずDBへ保存すること。

保存対象：

* meter_id
* timestamp
* value
* confidence
* image_path
* model

DBは現フェーズでは SQLite を使用する。

---

## Backend API設計ルール

最低限、以下のAPIを持つこと：

POST:

/api/v1/images

用途：

画像受信 → 推論 → DB保存

---

GET:

/api/v1/readings/latest

用途：

最新値取得（React表示用）

---

GET:

/api/v1/readings

用途：

履歴取得（グラフ用）

---

## React Frontend 実装ルール

最低限の機能：

* 最新値表示
* 履歴グラフ表示

データは必ず Backend API から取得すること。

推論処理をFrontendに実装してはいけない。

---

## 通知機能ルール

閾値を超えた場合：

Slack Webhook に通知する。

Webhook URLは：

環境変数で管理すること。

例：

SLACK_WEBHOOK_URL

コードに直接書いてはいけない。

---

## Codex実装制約（重要）

Codexは以下を守ること：

禁止：

* 大規模リファクタ
* 不要な依存追加
* ディレクトリ構造の変更
* API仕様の無断変更

必須：

* 最小差分で実装
* 既存コードを優先して再利用
* 不明点は実装前に質問する

---

## 実装手順ルール

Codexは必ず以下の順序で作業すること：

1.

実装計画を提示

内容：

* 変更ファイル
* 追加ファイル
* API変更
* DB変更

2.

ユーザー承認後に実装

---

## デモ優先原則（最重要）

現在はプロダクト完成ではなく

PoCフェーズである。

したがって：

最もシンプルで
最も早く
最も確実に動く方法

を選択すること。
